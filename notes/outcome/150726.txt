## Areas to improve

1. Encapsulation
- Internal data is still too exposed.
- In strict OOP, you should protect object state and control access through methods.

2. Single Responsibility Principle
- `Email` is doing too much at once: storing data, formatting, and changing state.
- `Inbox` also seems to mix storage and interaction logic.

3. Clearer object boundaries
- Each class should have one main responsibility and not overlap too much with others.

4. Validation and error handling
- Empty inbox, invalid index, missing sender/receiver, and bad state transitions should be handled more deliberately.

5. Better use of type clarity
- Parameters and return values should be more explicit and consistent.

6. Cleaner state design
- A simple boolean like `read` is fine for learning, but strict design often benefits from clearer state handling.

7. Naming and consistency
- Keep naming consistent and intuitive across the whole design.

8. Separation of behavior and presentation
- Printing and formatting should be treated differently from core object logic.

## Best way to approach this before coding further

Focus on these concepts first:

- Class vs object
- Encapsulation
- Composition vs inheritance
- Responsibility of each class
- Data hiding and controlled access
- When a method should belong to a class
